ClawdBot (OpenClaw) — Deep Dive

A comprehensive technical analysis of the multi-channel AI personal assistant framework

Version 2026.1.24-3 · MIT License · GitHub

What is ClawdBot?

In plain English: ClawdBot is an AI assistant that lives on your messaging apps — Telegram, WhatsApp, Discord, Signal, Slack, iMessage, and more. You text it like a friend, and it can read files, write code, search the web, control devices, and remember things across conversations.

The problem it solves: Most AI tools are separate apps you switch to, forget everything between sessions, and can't take actions. ClawdBot meets you where you already are (chat apps), remembers context persistently (via files on disk), and executes real actions (code, web, files, devices).

The Naming

  • ClawdBot — The runtime: the npm package you install and run as a daemon on your server.
  • OpenClaw — The open-source project: community, GitHub repo, ecosystem. Formerly "Moltbot."

What is a WebSocket? Why use it here?

HTTP is request-response: client asks, server answers, connection closes. WebSocket is a persistent, bidirectional connection — both sides can send messages anytime. Think of HTTP as letters vs WebSocket as a phone call.

Why here? ClawdBot needs real-time communication between adapters and the core. Messages must reach the agent instantly; response chunks must reach channels instantly. WebSocket provides this always-on, low-latency pipeline.

What is Lane-Based Concurrency?

A lane is a bounded queue. ClawdBot uses two nested lanes:

  • Session Lane (1 per session) — Messages in one chat process in order.
  • Global Lane (4 total) — Only 4 agent runs across all sessions simultaneously.

Nesting: enqueueSession(() => enqueueGlobal(async () => { ... })). Per-session ordering AND global resource control.

The Agent Loop

Step-by-Step: What Happens When You Send "What's the weather?"

  1. Channel receives — grammY gets the Telegram message, extracts text/media/metadata.
  2. Gateway routes to session agent:main:telegram:dm:your-id.
  3. Queue enqueues — waits for session turn, then global slot.
  4. Model resolution — picks configured model, selects a non-rate-limited auth profile.
  5. System prompt assembly — concatenates: AGENTS.md, SOUL.md, USER.md, skills, tools, runtime info.
  6. Streaming begins — full prompt + history + message go to LLM via streamSimple().
  7. Tool call — model calls web_search, tool executes, results return to model.
  8. Model continues — formulates answer from tool results.
  9. Response delivered — formatted, chunked, sent back through Gateway to Telegram.

The pi-coding-agent SDK

ClawdBot uses @mariozechner/pi-coding-agent and @mariozechner/pi-ai SDKs for the core agent loop: sending prompts, streaming responses, parsing tool calls, managing history.

Why an SDK? Agent loops have subtle complexity: streaming parsing, tool call extraction, context management, retries. The SDK handles this, letting ClawdBot focus on channels, memory, skills, and identity.

What is a Sub-Agent? When and Why?

A sub-agent is a separate AI conversation spawned by the main agent with its own session and context window, running in parallel.

When to spawn one?

  • Parallelizable work: "Research 5 companies" → spawn 5 sub-agents.
  • Different model needed: main uses Opus, sub-agent uses cheaper Sonnet.
  • Context preservation: offload to a sub-agent rather than filling the main context window.

Isolation: Own session key, history, and context. Communicates via sessions_send/sessions_history. Max concurrent: 8.

Skills vs Tools

Tools = code-level functions the agent calls directly (read a file, run a command). Built into the runtime.

Skills = prompt-level instructions (SKILL.md markdown files) that teach the agent to accomplish tasks using existing tools. Example: "To check weather, use exec to run curl wttr.in."

Memory System

The Three Core Memory Files

ClawdBot uses plain markdown files as its memory — no proprietary database, just files you can read and edit yourself:

FilePurposeWho Writes It
MEMORY.mdLong-term curated facts — things the agent has learned and wants to remember: your preferences, project details, important decisionsThe agent maintains it
USER.mdYour profile — name, timezone, communication preferences, context the agent needs about youYou (during onboarding) + agent updates
SOUL.mdThe agent's personality, identity, and behavioral rules — its "soul"You customize it

Why three files? Separation of concerns. USER.md is about you (rarely changes). SOUL.md is about the agent's identity (set once, tweak occasionally). MEMORY.md is the working knowledge base (constantly updated). This lets you reset memory without losing personality, or change personality without losing memory.

SOUL.md — What is an Agent's "Soul"?

SOUL.md defines the agent's persistent personality:

  • Personality: Formal or casual? Verbose or terse?
  • Identity: Name, self-reference style
  • Constraints: Things it should never do, sensitive topics
  • Style: Greetings, error handling, emoji usage

Injected into every system prompt for consistent behavior across all conversations.

Hybrid Search: BM25 + Vector Embeddings

When the agent searches its memory, it uses two completely different search strategies and combines them:

BM25 — Keyword Search (30% weight)

BM25 (Best Matching 25) — a ranking algorithm from 1994, still the gold standard for keyword search (Elasticsearch, SQLite FTS5). It scores documents based on:

  • Term Frequency: How often search terms appear (diminishing returns)
  • Inverse Document Frequency: Rare words score higher ("portfolio" in 2/1000 docs beats "the" in 990/1000)
  • Document Length: Normalizes for size

When keywords win: Exact names, error codes, technical terms, IDs — "TSLA" finds exactly "TSLA".

Vector Embeddings — Semantic Search (70% weight)

Vectors are lists of numbers. An embedding is a neural network that converts text into a vector (768 numbers). Key insight: similar meanings → similar vectors. "I love dogs" and "I adore puppies" get nearly identical vectors despite sharing no words.

Similarity search: (1) Convert query to vector, (2) compare against all stored vectors using cosine similarity (measures angle between vectors: same direction = 1.0, opposite = -1.0). All vectors are cached.

When vectors win: Conceptual queries — "how did my investments do?" finds "portfolio returns" and "stock performance." ClawdBot uses EmbeddingGemma 300M locally on CPU via node-llama-cpp — fully private.

Why Hybrid? Combining Both

BM25 misses semantic matches ("car" won't find "vehicle"). Vectors struggle with exact names and codes. mergeHybridResults() combines both (70% vector, 30% BM25) — documents scoring high on both rise to the top.

SQLite Storage

Stored in SQLite (~/.clawdbot/memory/agent.sqlite) with sqlite-vec for vectors and FTS5 for full-text search. Why SQLite? Single file (easy backup), zero config (no server), embedded in process (no latency). Perfect for a personal, single-machine agent.

Provider System

Auth Profile Rotation

An auth profile is a named credential (e.g., anthropic:default, anthropic:backup-key). Multiple profiles per provider. Why rotate? When one key hits its rate limit, ClawdBot silently switches to another and puts the exhausted one on cooldown.

Supported Providers

Anthropic (Claude) OpenAI (GPT/Codex) Google (Gemini) GitHub Copilot AWS Bedrock MiniMax Venice Qwen Portal Chutes OpenCode Zen Any OpenAI-compatible endpoint

Failover Chain

When a request fails, ClawdBot follows a cascade:

  1. Retry — transient errors get retried
  2. Rotate auth profile — switch API key, same provider
  3. Downgrade thinking — try without extended reasoning
  4. Fallback model — switch to different model
  5. Context compaction — summarize older messages, retry

Model Selection

Configured via agents.defaults.model.primary. Normalizes names across providers, checks context window guards, supports per-agent overrides. Dynamic catalog from pi-coding-agent SDK.

Channel Architecture

Each channel is an independent adapter translating platform protocols to ClawdBot's internal format via WebSocket:

  • New platforms only need a new adapter — core doesn't change
  • Platform features handled in adapter (reactions, buttons, typing indicators)
  • Formatting adapts per channel (no tables on WhatsApp, etc.)

Gateway as Control Plane

The Gateway is also the control plane: restart agent, apply config, self-update, manage sessions, monitor status. Supports Tailscale for secure remote access.

Key Differentiators

What makes ClawdBot unique compared to frameworks like LangChain, CrewAI, or AutoGen?

It's a Product, Not a Library

Most frameworks give you building blocks — you build UI, deployment, memory yourself. ClawdBot is complete: install, configure, message. Already running on your chat apps.

Persistent Identity Across Sessions

SOUL.md + AGENTS.md + MEMORY.md give continuous identity across sessions. Remembers projects, preferences, ongoing work. Git version-controlled.

Lane-Based Concurrency

Session serialization + global bounds. No race conditions, predictable under load. Rare in open-source frameworks.

Hybrid Memory Search

BM25 + vectors, fully local option. No Pinecone/Weaviate needed, full privacy.

Skill Marketplace

ClawdHub: community skills discovered and installed from within a conversation. Agent extends itself.

Proactive via Heartbeat + Cron

Heartbeat polls + cron jobs enable proactive behavior — email, weather, calendar checks without being asked. Not just a reactive chatbot.

Device Nodes

Companion apps on macOS/iOS/Android give access to camera, screen, notifications, location — bridging digital and physical.
